# 回放端 `new_structured_capture` 配色 / 绕序修复报告

> 日期：2026-06-18
> 场景：`example/new_structured_capture`（Unity URP 抓帧：天空大气 + 地面 + 中央 6 个方块）
> 入口：`guiReplayer` / `testReplayCapture`（同一套 `ReplayEngine` + `ResourceManager`）
> 调试工具：`REPLAY_DUMP=<ppm>`、`REPLAY_STOP_AT=<eid>`、`REPLAY_DEBUG_TEX=<id>`（均在 `testReplayCapture.cpp`，环境变量门控）

---

## 0. 现象起点

接上一阶段（深度测试、UBO、sampler 修复之后），中央的 6 个方块渲染成**纯黑剪影**——原本应是一排**灰色**方块。逐步排查，最终修复后方块与地面同为中性灰，天空、地面亮度正确。

本 session 共定位并修复 **4 个独立根因**，全部集中在 `ResourceManager::uploadTextureData` / `restoreState` 与 `Command.cpp` 的 `FrontFaceCommand`。核心矛盾都是同一类："抓帧只存了部分初始化态 / 用了 WebGL·Unity 的隐含约定，回放端在 desktop GL core profile 下必须显式重建或补偿。"

---

## 1. 调试链条（按发现顺序）

### 问题 1 ── 方块纹理整体采样为黑（mip 不完整）

**定位过程**
1. `REPLAY_DUMP` 导出帧 → 6 个方块纯黑，仅最右方块右侧有一丝灰（用户观察）。
2. 读 `commands.json`：program 4（shader 8，Unity URP Lit）画 6 个方块，采样 `_BaseMap`(2D)、3 个 `samplerCube`、阴影图 `sampler2DShadow`。
3. 加 `FIXUP_DEBUG` 打印每个 sampler 的 `chosen unit`：确认 sampler 修复**正确**（cube→u0=tex9，`_BaseMap`→u3=tex1，shadow→u5=tex4），**排除 sampler 问题**。
4. 用户提示："hack shader 8 让 `SV_Target0` 输出固定黄色 → 方块变黄，证明几何/材质没问题，是光照/资源问题。"
5. 检查 shader 8 用到的纹理元数据：**每张纹理 `parameters: {}`**，且 `levels:1`、`mipsGenerated:false`。

**根因**
抓帧不记录 `gl.texParameter`（属初始化调用）。回放只上传 level 0，纹理便保留 GL 默认的
`GL_TEXTURE_MIN_FILTER = GL_NEAREST_MIPMAP_LINEAR` —— 该过滤需要完整 mip 链，单层纹理因此
**mip 不完整 → 所有 `texture()`/`textureLod()` 返回 (0,0,0,1) 黑**。在 Lit 着色器里这会把
`albedo = _BaseColor × texture(_BaseMap)`、`SH·albedo`、IBL 全部乘成 0，只剩与 albedo 无关的
电介质高光（F0≈0.04）—— 那一丝灰就是它。

**解决**（`uploadTextureData`，采样纹理分支）
上传 level 0 后 `glGenerateMipmap` + 三线性 / `CLAMP_TO_EDGE` 默认参数；渲染目标用 `GL_LINEAR`。
抓帧若有 `m_parameters` 仍后置覆盖默认值。

**效果**：方块正面 **(10,12,16) → (147,163,187)**，由黑变灰（偏白/偏蓝，见问题 2/3/4）。

---

### 问题 2 ── 方块"发白"，灰度远亮于地面（HDR 探针被当 LDR 解码）

**定位过程**
1. 方块变灰后偏白偏蓝，与地面灰差距大。
2. 读 UBO：`unity_SpecCube0_HDR.x = 4.59`、`_GlossyEnvironmentCubeMap_HDR = 4.59`（着色器对 cube 反射做 `color.rgb × 4.59` 的 HDR 解码）。
3. 实验 `CUBE_SCALE=0.22`（≈1/4.59）把 cube 像素整体压暗 → 方块 **(39,44,53)** ≈ 地面 **(36,32,30)**，定位 lever。
4. `0.22` 恰好接近 sRGB→linear 的衰减，遂试 `CUBE_SRGB=1` 把 cube 当 sRGB 上传 → **(38,47,62)** 同样匹配地面。

**根因**
Unity 反射探针是 HDR cube，抓帧用 `FBOReadPixelsCubeMap` 存成 RGBA8 —— 即 sRGB 编码的**显示值**，
而非原始线性/编码辐射度。回放把它当线性 RGBA8 采样后再 ×4.59，导致间接光**过亮约 4-5 倍**，
冲淡了应有的方向光，使方块呈扁平的亮蓝探针色。

**解决**（`uploadTextureData`）
RGBA8 cube map 以 `SRGB8_ALPHA8` 上传，采样时先 sRGB→linear，HDR 解码后落入正确量级。

**效果**：方块 **(147,163,187) → (38,47,62)**，与地面灰接近。

---

### 问题 3 ── 最后一个 LinearToSRGB pass "执行后画面无变化"（色彩管线被抵消）★用户关键线索

**定位过程**
1. 用户观察：最后一个 pass（prog2 / shader 4）是 `fragColor = LinearToSRGB(texture(tex,uv))`，但执行前后画面无差别。
2. `REPLAY_STOP_AT=163`（最后一 pass 前）vs `=174`（之后）逐像素对比：**完全相同**
   —— cube (38,47,62)→(38,47,62)、sky (78,117,179) 不变。确认 LinearToSRGB **是个空操作**。
3. 检查 tex2（最终颜色 RT）：`internalFormat = 35907 = SRGB8_ALPHA8`。

**根因**
回放从不开 `GL_FRAMEBUFFER_SRGB`，Unity 着色器**手工**做 gamma（最后一 pass 显式 LinearToSRGB）。
若把颜色 RT 按 sRGB 格式分配，GL 在采样时**自动 sRGB→linear 解码**，正好与着色器的 LinearToSRGB
**编码相互抵消** → 最后一 pass 不改变任何东西，屏幕显示的是**原始线性值**（偏暗、相对偏蓝，即"发蓝"）。

**解决**（`uploadTextureData`，渲染目标分支）
把 sRGB 颜色 RT 按线性等价格式分配：`SRGB8_ALPHA8 → RGBA8`、`SRGB8 → RGB8`，
让手工 LinearToSRGB 成为唯一的一次 gamma 转换。

**效果**：最终输出 **(38,47,62) → (108,119,135)**，gamma 正确、明显提亮、蓝偏减弱；
方块灰与地面灰开始接近。

---

### 问题 4 ── 最右方块"被遮挡的右面渲染到了前面"（绕序反向）★用户关键线索

**定位过程**
1. 用户观察：透视上看不到的最右方块右面却出现了，疑似"被挡住的面渲染到前面"。
2. 排查深度：FBO3 有深度附件 tex8（DEPTH24_STENCIL8），EID62 清深度，`restoreState` 已按"有深度附件则开 `GL_DEPTH_TEST`"补偿。`REPLAY_STOP_AT=134 REPLAY_DEBUG_TEX=8` 读回深度：min=0.986、mean=0.9999 → **深度确实在写、自遮挡正常**，排除深度。
3. 读 `ObjectToWorld`：6 个方块全为**单位旋转**（轴对齐），仅 x 平移不同；相机在 x=0，方块 x∈[-5.55,+2.33]。逐方块采样边缘：只有屏幕最右（cube4, x=+2.33）右侧多出 6px 宽、(127,135,145) 的**独立面**，其余 5 个干净。
4. 决定性实验 `FLIP_WINDING=1`（翻转 `glFrontFace`）：**多余的面消失**（最右方块变均匀），且所有方块更**中性**（114,115,114 vs 之前 108,119,135）。

**根因**
Unity 以**左手**裁剪空间约定其矩阵与网格绕序；把这些矩阵原样喂给**右手** desktop GL，
有效三角形绕序被反转。沿用抓帧的 `GL_CCW` 时，背面剔除剔掉了正面、**保留背面** →
凸方块自遮挡反向（被挡的远面盖住近面），且被看到的是**背向太阳的暗面**（仅 GI 照亮 → 扁平偏蓝）。
这也解释了为何"发蓝"在翻转绕序后进一步减弱。

**解决**
在 `restoreState` 与 `FrontFaceCommand::execute` 两处翻转 front-face：
`glFrontFace(captured==GL_CCW ? GL_CW : GL_CCW)`（两处都改，防止帧内 frontFace 命令把补偿覆盖回去）。

**效果**：最右方块多余面消失（115,117,115 均匀）；方块整体由偏蓝转为中性灰，正面被太阳正确照亮。

## 2. 最终效果

| 采样点 | 修复前 | 全部修复后 |
|---|---|---|
| 方块正面 | (10,12,16) 纯黑 | **(115,117,115)** 中性灰 |
| 地面 | (36,32,30) 偏暗 | **(105,98,93)** 灰 |
| 天空 | 偏暗 | **(135,164,202)** 均衡蓝 |
| 最右方块右缘 | 多出亮面 | 均匀，无多余面 |

方块灰 ≈ 地面灰，无多余面，gamma 正确，配色中性——与抓帧参考一致。`No GL errors`。

## 4. 经验沉淀（与 [工作报告-回放端修复] 一脉相承）

"partial capture" 缺的初始化态 / 隐含约定，必须在回放端重建或补偿，且这些坑在 GL core profile 下只表现为黑、暗、扁平、自遮挡反向，而非报错：

- **纹理参数从不被抓**（`parameters:{}`）：默认 min filter 需要 mip → 单层纹理 mip 不完整 → 采样为黑；HDR 探针按显示值存储 → 需按 sRGB 采样才能配合 ×HDR 解码。
- **颜色空间**：回放不开 `GL_FRAMEBUFFER_SRGB` 且着色器手工 gamma → 颜色 RT 必须是线性格式，否则末端 LinearToSRGB 被自动解码抵消。
- **绕序**：Unity 左手约定的矩阵在右手 GL 下绕序反转 → front-face 必须翻转。

> 注：绕序翻转目前对所有抓帧全局生效，适用于本工具针对的 Unity URP 抓帧；若日后回放真正 `GL_CCW` 的非 Unity 抓帧需改为按投影手性判定。
