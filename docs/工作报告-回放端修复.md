# OpenGL 回放端修复工作报告

> 项目：WebGL2 抓帧 → 桌面 OpenGL 回放端
> 范围：代码 review、Bug 修复、frame6 复杂场景渲染调试
> 测试样例：`example/20260615_110825_frame0`（简单立方体）、`example/20260611_161020_frame6`（复杂 URP 场景）

---

## 1. 概述

回放端的设计：Reader 从 capture 读入资源元数据与帧开始时的初始状态，将 GL 状态调到与 capture 一致，然后通过 `Command` 虚函数逐条重放 WebGL2 帧内命令。命令流**只包含帧内命令，不含 WebGL 初始化命令**，因此初始化状态必须由其他途径重建。

修复前现状：frame0 画面空白、frame6 完全无输出。修复后：

| capture | 命令数 | GL 错误 | 渲染结果 |
|---|---|---|---|
| frame0 | 10 | 0 | 彩色立方体 ✓ |
| frame6 | 170 | 0 | 天空大气 + 中央 6 个方块 ✓ |

---

## 2. 问题与解决（按类别）

### 2.1 P0 — 直接导致无法回放

**P0-1 初始化流程漏步骤（画面空白根因）**
- 回放驱动手写了一遍上传流水线，**漏掉 `restoreVertexArrays()` / `restoreFramebuffers()`**，只到 `restoreState()`。而 VAO 顶点属性、FBO 附件的恢复正是由那两个函数负责。
- 后果：VAO 从未配置 → `drawElements` 无顶点数据 → 空白。
- 解决：驱动改为调用一站式入口 `ResourceManager::uploadAllResources()`（顺序：buffers→textures→shaders→programs→restoreState→restoreFramebuffers→restoreVertexArrays）。

**P0-2 `WORKING_DIR` 改错文件**
- CMake 用 `config/config.in` 生成根 `config.h`，但测试 `#include "../src/config.h"`（另一份硬编码 `C:/Rep/...`）。默认路径指向不存在目录，加载直接失败。
- 解决：测试改用 CMake 注入的 `PROJECT_ROOT_DIR`，删除对 `src/config.h` 的依赖。

**附带：阻塞编译的预存 Bug**
- `std::regex::multiline` 在该工具链 libc++ 不支持，导致整库编译失败。两处正则本就手写行上下文判断，去掉 `multiline`/`^`、改用 `\b`，行为等价。([ShaderInterpreter.cpp](../src/shaderTranslation/ShaderInterpreter.cpp))

### 2.2 P1 — 正确性 Bug

**P1-1 `getMappedHandle` 未命中即抛异常 → 崩溃**
- 大量 `execute()` 直接调用 `getMappedHandle`，缺失 id 时 `throw std::runtime_error` 未捕获，整帧终止。
- 解决：新增 `getMappedHandleOr(kind, id, fallback=0)`（[UniversalDMapper.h](../src/mappers/UniversalDMapper.h)），绑定类命令改用它；上传/编译/链接类命令加 `hasMappedHandle` 守卫跳过。

**P1-2 `bufferSubData` 被当成 `bufferData`**
- 原来永远调 `glBufferData`（重新分配整个 buffer）、忽略 offset、强制 `GL_DYNAMIC_DRAW`。
- 解决：`bufferSubData` 走 `glBufferSubData(target, offset, size, data)`；inline 数据按 double→float 转换。

**P1-3 `texSubImage2D` 参数错位 + 走了 `texImage2D`**
- 两者 WebGL 参数布局不同（subImage 有 xoffset/yoffset、无 internalFormat/border），原解析器用同一套下标导致错位。
- 解决：拆分解析，subImage 用 `glTexSubImage2D` 并新增 xoffset/yoffset 字段。

**P1-4 uniform 值的两个致命问题**
- ① 数据以 `double` 存储，却用 `reinterpret_cast<const GLfloat*>` 当 float 读 → 矩阵全是垃圾。改为逐元素 double→float 转换。
- ② `_snapshot` 机制把所有 uniform（含 `mat4`）平铺成 `uniform4fv`，对 mat4 调 `glUniform4fv` → `GL_INVALID_OPERATION (0x502)`。解决：用 `glGetActiveUniformsiv(GL_UNIFORM_TYPE)` 查真实类型，分派到正确的 `glUniformMatrix4fv`/`glUniform*fv`。

**P1-5 顶点属性 location 不匹配**
- 翻译后的 GLSL 顶点输入无 `layout(location=)`，链接器随意分配，与 VAO 的属性 index 不一致。
- 解决：link 前用 `glBindAttribLocation` 按名字绑定 location，名字来源于 vaos + state + `vertexAttribPointer` 命令的并集。

**P1-6 `currentSamplers` 未解析** — `parseState` 现在解析 sampler 绑定。

### 2.3 P2 — 健壮性 / 多帧

- **P2-1 主循环每帧重放 create/delete → 泄漏**：`CreateResourceCommand` 改为幂等（已存在映射则复用，不再每帧 `glGen` 新句柄）。
- **P2-2 切换 capture 不重置全局 mapper**：`UniversalDMapper::clear()` + `ResourceAllocator::deleteAllResources(capture)`（删 GL 对象 + 清映射，含合成的默认 VAO），测试退出时调用。
- **P2-3 EBO 上传污染默认 VAO**：buffer 数据统一经 `GL_COPY_WRITE_BUFFER` 上传，不动任何 VAO 的绑定。

### 2.4 核心 profile 相关

- **核心 profile 没有可用的默认 VAO**：WebGL 自由使用 VAO id 0，而核心 profile 下用 VAO 0 调 `glVertexAttribPointer`/绘制会 `GL_INVALID_OPERATION`。解决：`ensureDefaultVertexArray()` 为 id 0 生成真实 VAO 并在 init 期间保持绑定；所有绑定走 `getMappedHandleOr(VertexArray, id)`。
- **Cube map** 必须逐面定义（`glTexImage2D(GL_TEXTURE_CUBE_MAP,…)` 是 `GL_INVALID_ENUM`）：用单张 capture blob 填充 6 个面（best-effort）。
- **按真实 target 绑定纹理**：`currentTextures` 只有 unit→id，把 cube/depth 纹理绑到 `GL_TEXTURE_2D` 会 `0x502`；`restoreState` 现按 `TextureWrapper::m_target` 绑定。
- **buffer `usage` 可能为 0** → `glBufferData(...,0)` 是 `GL_INVALID_ENUM`，缺省 `GL_STATIC_DRAW`。

## 3. frame6 渲染问题深度调试（重点）

frame6 是 Unity URP 场景：170 命令、69 buffer、13 纹理、3 个 FBO、6 个 program、UBO、cube map、阴影贴图。修复 2.x 后**天空渲染出来了，但中央的 6 个方块缺失**。下面是逐步定位与解决的完整过程。

### 帧结构
- **FBO2**：阴影 pass（program 3，4096/2048 quadrant，12 次 drawElements）→ 深度贴图 tex4。
- **FBO3**：主场景（program 4 画 6 个方块 + program 5 画天空 5040 顶点）→ 颜色 tex6。
- **FBO1**：后处理（program 6 全屏三角）→ tex2。
- **FBO0**：最终合成（program 2 采样 tex2）→ 屏幕。

### 问题 1：渲染目标纹理尺寸错误（导致整体平坦）
- **现象**：最初 frame6 输出一片平坦深色。
- **定位**：FBO 附件纹理（tex2/4/6/8）按降采样的 `capturedWidth` 分配（如 4096 阴影图被分配成 1024），帧内各 pass 渲染进过小的目标 → 整条多 pass 链崩溃。
- **解决**（[ResourceManager.cpp](../src/resourceManagement/ResourceManager.cpp) `uploadTextureData`）：从 `framebuffers.json` 收集所有附件纹理 id，对这些**渲染目标按完整 `width×height` 分配（null 数据，帧内自己渲染填充）**；只有被采样的纹理才用降采样 blob。深度/模板 RT 若 `format/type==0` 则由 internalFormat 推导。
- **结果**：天空大气场景正确渲染出来。

### 问题 2：Uniform Block 绑定缺失（UBO 串味）
- **背景**：WebGL 的 `gl.uniformBlockBinding()` 是初始化调用，**不在帧命令流里**。
- **现象**：方块的 `UnityPerDraw`（变换矩阵）和 `UnityPerMaterial`（材质）两个 block 默认都绑到 binding 0，互相串味。
- **解决**（`linkProgram`）：link 后令每个 block 的 binding = 其 block index（`glUniformBlockBinding(prog, i, i)`）；capture 的 `bindBufferBase/Range` 的 binding point 顺序与 block 声明顺序一致，故能对上（验证：UnityPerDraw=block0→point0→buf68，UnityPerMaterial=block1→point1→buf59）。

### 问题 3：非数组 uniform 的 `[0]` 后缀（光照丢失 → 纯黑）
- **现象**：方块光栅化成功（occlusion query 实测 2601 个 sample 通过），但片元输出**纯黑**（tex6 HDR max=0）。
- **定位**：`_MainLightColor` 等是非数组 `uniform vec4`，但 capture 命名为 `_MainLightColor[0]`。`glGetUniformLocation("_MainLightColor[0]")` 对非数组返回 -1 → 被静默跳过 → 光照为 0。
- **解决**（[Command.cpp](../src/drawResources/Command.cpp) `getUniformLocationFlexible`）：查询失败且名字以 `[0]` 结尾时，去掉 `[0]` 用基名重试。三个 uniform 命令都改用它。

### 问题 4：Sampler → 纹理单元绑定缺失（贴图读 0）
- **背景**：sampler 的 `gl.uniform1i` 也是初始化调用，**不在帧里**。所有 sampler 默认指向 unit 0。
- **现象**：方块 pass 期间 unit 0 上绑的是 cube map（tex9），而 `_BaseMap` 是 `sampler2D` → 读 cube → 返回 0 → albedo=0 → 黑。
- **关键难点**：单元的"类别"在不同 pass 间会变（全屏后处理 pass 又把 unit 0 绑成 2D），所以**初始化时一次性指定会破坏后处理 pass**（实测会导致整屏变白）。必须**逐 draw 动态修正**。
- **解决**（`fixupSamplerBindings`，每次 draw 前调用）：只在 sampler 当前单元绑定的纹理"种类不匹配"时，把它重定向到一个当前绑定了匹配种类纹理的单元（2D / cube / depth-shadow）。depth 纹理通过 `Command::registerDepthTexture` 预注册，使 `sampler2DShadow` 能路由到深度单元。
- **结果**：方块拿到 albedo，开始写入（但此时仍很暗——只有环境光）。

### 问题 5：`GL_DEPTH_TEST` 未开启（阴影图空 + 天空盖住方块）★最终关键
- **现象**：方块只有微弱环境光（HDR ~0.00036），主平行光贡献为 0。
- **定位**：`REPLAY_DEBUG_TEX=4` 读回阴影贴图 tex4 → **全 1.0（空的）**。原因：`state.capabilities` 为 `{}` 且帧内无 enable 命令，`GL_DEPTH_TEST` 全程关闭 → 阴影 pass 不写深度（阴影图空）→ 主光被判为全阴影；同时无深度测试时后画的天空会直接覆盖方块。
- **根因**：capture 没有记录初始化时的 `gl.enable(DEPTH_TEST)`。
- **解决**（`restoreState`）：若 `capabilities` 未列出 `GL_DEPTH_TEST`，但帧内存在带深度/深度-模板附件的 FBO，则开启 `GL_DEPTH_TEST`。
- **结果**：阴影图写入正确深度（min=0.436），方块作为逆光剪影出现在中央——**6 个方块成功渲染**。

### frame6 调试链小结
渲染目标尺寸 → UBO 绑定 → uniform `[0]` → 逐 draw sampler 修正 → 深度测试重建，五步缺一不可。其中"初始化态缺失"（UBO 绑定、sampler 绑定、深度测试开关）是 partial capture 的共性缺口，需在回放端重建。

## 4. 调试工具（环境变量门控，默认关闭）

均在 [testReplayCapture.cpp](../test/testReplayCapture.cpp) / [ResourceManager.cpp](../src/resourceManagement/ResourceManager.cpp)：

- `REPLAY_DUMP=<path.ppm>`：导出 framebuffer（PIL 转 PNG 查看）。
- `REPLAY_STOP_AT=<eventId>`：在该命令后停止，把当前 draw FBO blit 到屏幕并报告 HDR 浮点最大值——用于逐 pass 隔离。
- `REPLAY_DEBUG_TEX=<captureId>`：回读某深度纹理的 min/max/mean——定位阴影图是否有内容。
- 每阶段 / 每命令 `glGetError`（`dbgGlPhase` / `checkGlErrors`）：无错误时静默。

这些工具是定位 frame6 各问题的关键，正常运行不产生副作用。

## 5. 已知限制 / 后续

- frame6 方块呈深色逆光剪影：对低角度逆光场景物理上合理，且位置/数量正确；但缺少原始 WebGL 参考图，无法逐像素校验阴影偏移 / 反射探针等细节保真度。
- Cube map 6 个面共用单张降采样 blob（capture 只存一张），反射/天空盒细节非精确还原。
- Sampler / UBO 绑定、深度测试开关均为基于 capture 信息的启发式重建（因初始化命令未被抓取）。如后续扩展 capture 抓取这些初始化状态，可去掉启发式、提升保真度。
